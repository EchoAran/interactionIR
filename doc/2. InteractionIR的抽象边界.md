InteractionIR的设计应该不限于human-AI interaction，核心概念完全适用agent-agent interaction。maybe这个interaction skill能集成到多智能体系统中支持多agent交互，agent交互质量影响multi-agent任务完成。

InteractionIR 的抽象边界应当从 human-AI interaction 提升到 party-party interaction，其中 party 可以是 human、agent、tool proxy、service endpoint。文档里已经把交互拆成 slots、policies、dialogue acts、tool recipes、checkpoints，这些概念天然支持 agent-agent 场景；近年的多智能体框架与综述也基本沿着这个方向发展。AutoGen 直接把多智能体应用建模为通过消息交换完成任务的 conversable agents；MetaGPT把 SOP 编进多 agent 协作流程，用中间结果校验来减少错误；2025 年的 communication centric survey 更明确把 LLM-MAS 定义成 communication protocol constrained automated system；另一个系统综述把 mutual interaction 列为 LLM-based MAS 的五个核心组成之一。([微软 GitHub][1])

所以，从研究建模上看，InteractionIR 的核心对象并非“人机对话”本身，核心对象应当是“带角色、带状态、带约束、带证据要求的交互过程”。文档里强调的“controller 而非 flowchart”“约束 allowed acts 而非固定脚本”“用 checkpoints 和 provenance 控制生成”，放到 agent-agent 交互里依旧成立。差别主要落在 specialization 层：human-AI 需要处理意图澄清、偏好协商、问答预算、用户体验；agent-agent 需要处理协议兼容、可验证消息结构、委托与承诺、冲突消解、可靠性与安全。 ([arXiv][2])

InteractionIR is a role and protocol aware communication IR for heterogeneous parties。这样 human-AI interaction 只是一个 profile，agent-agent interaction 是另一个 profile。这样做有两个直接好处。第一，当前的 IR 组件几乎不用推倒重来。第二，它能自然接进多智能体系统，把“interaction skill”从单技能编译产物变成多 agent runtime 的公共协调层。

在这个扩展视角下，InteractionIR 至少要新增五类一等公民概念。

第一类是 party model。当前文档里的 dialogue acts 默认面向用户。扩展后应当给每个参与方声明 role、capabilities、authority、trust level、cost profile、visibility scope。多 agent 协作里，谁能提任务、谁能调用工具、谁能提交最终结论、谁只能提供证据，这些都需要进入 IR。MetaGPT 的角色分工与 SOP 编排已经证明，角色和流程约束会显著影响协作质量。([arXiv][3])

第二类是 message contract。human-AI 可以容忍自然语言含混，agent-agent 交互需要更强的结构化消息。建议把 ASK、PROPOSE、REQUEST_SOURCES、CONFIRM 这类 dialogue acts 再细化成 machine checkable act schema，例如 propose_plan、delegate_task、request_evidence、submit_result、challenge_claim、accept_commitment、handoff、escalate。每种 act 都带输入槽位、输出槽位、预条件、后置条件和 evidence binding。协议综述已经把 inter agent protocols 单独分类，并强调效率、可扩展性、安全性、可靠性等维度；这正说明 agent-agent 交互需要比人机对话更严格的消息契约。([arXiv][4])

第三类是 commitment state。多 agent 任务失败，很多时候问题不在推理能力，问题在承诺丢失、交接模糊、重复工作、冲突覆盖。InteractionIR 里应当加入 commitment、responsibility、deadline、dependency、verification status 这类状态对象。这样 runtime 才能知道某个 agent 当前是在“请求协助”“接受委托”“等待证据”“提交可验证结果”还是“发起争议”。Anthropic 在生产多 agent research system 时专门提到 coordination、evaluation、reliability 是新增挑战，这类状态层就是解决这几个问题的关键抓手。([Anthropic][5])

第四类是 communication economics。agent-agent 交互的成本通常高于 human-AI 单轮问答，因为会带来 token、延迟、上下文膨胀和协作回路。文档里已有 question budget、max_rounds_without_progress，这个思想应扩展为 communication budget，包括 max_parallel_branches、max_delegation_depth、max_revision_cycles、max_evidence_per_claim、latency_budget、token_budget。2025 年关于 agent protocols 的综述把效率、延迟、可扩展性列为核心评估维度，这与 InteractionIR 的预算机制高度一致。([arXiv][4])

第五类是 security and trust policy。human-AI 交互里强调 no invention 和 provenance。到了 agent-agent 场景，还要加 agent identity、capability attestation、message authenticity、cross org boundary policy、sensitive data handling、tool invocation permission。2025 年 agent communication security survey 明确把 agent-agent communication 作为独立阶段，并指出跨组织代理协作会扩大攻击面，带来隐私泄露、agent spoofing、DoS 等风险。([arXiv][6])

所以，“interaction skill 能集成到多智能体系统中支持多 agent 交互”这个想法很有价值。我会把它表述成：interaction skill 不应仅仅是某个技能仓库中的一项 skill，它更适合作为 multi agent runtime 的 meta skill 或 coordination substrate。AutoGen 的 conversable agents、MetaGPT 的 SOP 编排、Anthropic 的多 agent research system，都在用各自方式实现这个协调层，只是它们目前大多停留在 framework convention、prompt pattern 或 runtime logic，还缺少一个统一、可编译、可验证、可评测的 InteractionIR。([微软 GitHub][1])

从现有文档继续往前推，最自然的系统架构是这样。底层保留 CapabilityIR、BindingIR、DataIR、PolicyIR。其上增加一个通用 InteractionIR，负责 party model、act schema、slot state、commitment state、evidence contract、budget、checkpoint。人机交互技能和多 agent 协作技能都编译到这一层，只是 profile 不同。再上面是 runtime controller，依据当前 stage 和 state 计算 allowed acts、required evidence、next queries、delegate targets、conflict resolution。文档里已经有 decision API 和 Rego policy 模块，这一层几乎可以直接扩展。

这个命题接下来可以被操作化成一套评测。至少测五项：task success rate、coordination overhead、invalid handoff rate、unsupported claim rate、repair efficiency。前两项衡量任务完成与通信成本，第三项衡量交接质量，第四项衡量证据约束，第五项衡量系统在冲突和失败后的恢复能力。文档里的 checkpoints、required evidence、question budgets、no invention policy，已经给这些指标提供了实现基础。 ([arXiv][4])
