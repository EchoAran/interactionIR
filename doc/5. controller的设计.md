按现在的分层，LLM-driven policy-guided interaction controller 应当被设计成一个“面向 InteractionIR 的受约束状态更新器与动作选择器”。它持续读取当前 InteractionIR、生成状态补丁提案、做策略判定、选择下一语义动作，再把该动作交给同一个或另一个 LLM 实例去表述。这个设计和文档里“interaction 是 stateful information model，运行时循环是 update state → policy check → action selection → optional tool orchestration → checkpoint gating”的主线一致。 A2A 也把 task 建模为有生命周期的 stateful object，并把 message、artifact、task state 分离；MCP 把 lifecycle、tools、resources、elicitation 分开定义；OPA 则明确采用“输入结构化数据，返回策略决策”的模式，这三者共同支持“状态中间件 + 控制层”的分层。([A2A Protocol][1])

把 controller 压成六个内部部件，外加一个统一回合循环。

第一部分叫 `State Interpreter`。它只做一件事：把最新用户输入、工具结果、上一轮 act、当前 checkpoint 翻译成 `InteractionIR patch proposal`。这里输出的东西不是最终写库结果，只是候选补丁。需求访谈场景下，它主要负责识别 slot 值更新、冲突暴露、新 issue、新 assumption、checkpoint 候选推进。这样可以把“理解当前输入”与“决定下一步干什么”拆开，减少单次 LLM 调用同时承担太多职责。文档里反复强调 slot、acts、checkpoints 都属于交互状态层，这个解释器就是把自然语言映射到这些状态对象。

第二部分叫 `Patch Validator`。它接收候选补丁，检查三件事：结构是否合法，profile 允许不允许，是否违反硬约束。比如需求访谈 profile 允许新增 `assumption_slot`，允许把 `scope` 从 proposed 提升到 grounded，允许创建 `open_question` issue；但如果某补丁试图删除已 frozen 的 slot，或把无来源值直接写成 grounded，它就会失败。A2A 明确支持协议扩展和状态机扩展，这说明 schema 可演化是合理方向；OPA 又天然适合对 JSON 状态做不变量判断，所以补丁校验层很适合用 declarative policy 做。([A2A Protocol][2])

第三部分叫 `Policy Evaluator`。它基于“当前有效状态 + 已通过校验的候选补丁 + profile + runtime context”给出决策对象。这里的输出不直接是自然语言，输出应是结构化决策，例如：

```json id="e4u7cv"
{
  "decision": "require_input | allow | require_tool_query | require_review | block",
  "allowed_acts": ["ask_fact", "clarify_term", "summarize_understanding"],
  "required_updates": [],
  "required_evidence": [],
  "checkpoint_effect": {
    "enter": [],
    "exit": [],
    "hold": ["cp:context_seeded"]
  }
}
```

OPA 官方文档明确把策略决策定义成“对输入和数据求值后返回结论”，而且输入是任意结构化数据；这和 controller 需要的 decision packet 很贴近。([Open Policy Agent][3])

第四部分叫 `Act Planner`。它在 `allowed_acts` 集合里选下一语义动作。这里用 LLM 是合理的，因为需求访谈里“当前最该问什么”“先澄清术语还是先确认目标用户”这类选择具有强上下文性。文档里已经指出 interaction 无法预编译成 rigid tree，运行时应给 LLM 一个受限动作集，由它在动作集内选择微观推进方向。 这一层的输出建议长成：

```json id="m8q2rh"
{
  "chosen_act": {
    "act_type": "clarify_term",
    "about_slots": ["slot:problem_statement"],
    "goal": "澄清“平台”具体指内部后台还是面向客户产品",
    "priority": 0.84
  },
  "alternatives": [
    {"act_type": "ask_fact", "about_slots": ["slot:target_user"]},
    {"act_type": "summarize_understanding", "about_slots": ["slot:problem_statement","slot:primary_goal"]}
  ]
}
```

第五部分叫 `Response Realizer`。它把 `chosen_act` 渲染成最终给用户的话。这个部件只接收动作约束、slot 语义、风格约束、question budget，不直接读全量原始状态，从而减少跑偏。MCP 的 elicitation 明确允许在其他协议功能内部嵌套请求用户输入，而且不规定具体交互界面模型，这恰好说明“请求输入”可以是协议层允许的行为，由上层系统决定怎样呈现。([Model Context Protocol][4])

第六部分叫 `State Committer`。它负责把本轮已生效的 patch、act、artifact、checkpoint 写回 InteractionIR store，并生成 trace。这样 controller 的每一轮都会留下三样东西：收到什么，判断了什么，提交了什么。A2A 把 message、artifact、task state 分离，且任务具有清晰 lifecycle，这种提交方式很适合和外部协议做映射。([A2A Protocol][1])

把它们串起来，一轮标准控制循环可以定义成：

`observe → interpret → propose_patch → validate_patch → evaluate_policy → choose_act → realize_response → commit_state`

这条链里，LLM 主要驱动三个点：`interpret`、`choose_act`、`realize_response`。
policy 主要约束三个点：`validate_patch`、`evaluate_policy`、`commit_state`。
这样既保留 LLM 的适应性，又让状态维护和动作边界可检查。

如果要做成一个真实可实现的 controller，我建议它的输入和输出固定成下面两个对象。

控制器输入：

```json id="g5t9ls"
{
  "interaction_id": "int:req_interview:01J...",
  "profile_id": "requirements_interview/v1",
  "state": {
    "slot_state": [],
    "issues": [],
    "checkpoints": [],
    "recent_acts": [],
    "artifacts": [],
    "evidence": []
  },
  "runtime_context": {
    "last_user_message": "string",
    "tool_results": [],
    "available_tools": [],
    "budget": {
      "max_questions_this_turn": 2,
      "max_turns_without_progress": 2
    }
  }
}
```

控制器输出：

```json id="w7n4jk"
{
  "patch_proposal": {
    "ops": []
  },
  "decision": "require_input",
  "allowed_acts": ["clarify_term", "ask_fact"],
  "chosen_act": {
    "act_type": "clarify_term",
    "about_slots": ["slot:problem_statement"]
  },
  "response_plan": {
    "prompt_style": "concise_interviewer",
    "content_constraints": [
      "最多两个问题",
      "优先澄清高歧义术语"
    ]
  },
  "tool_plan": [],
  "commit_plan": {
    "write_act": true,
    "write_checkpoint": false
  }
}
```

这里最关键的是 `patch_proposal` 和 `chosen_act` 分开。因为在需求访谈里，经常会出现这种情况：当前轮用户输入已经足够让某些 slot 前进，但下一步动作仍需继续追问另一些 slot。也就是说，控制器先更新状态，再决定下一动作，不能把“状态理解”和“对话推进”混成一步。

接下来是策略层本身。为了让 controller 真正做到 policy-guided，我建议 policy 分成四组。

第一组叫 `State Integrity Policies`。
它只管状态写入合法性。例如：
`grounded slot 必须有 source_refs`
`frozen slot 不允许直接覆盖`
`controller 新增 slot 只能落在 profile 允许的 extension_policy 上`
`已 resolved issue 只能 reopen，不能静默删除`

第二组叫 `Interview Progress Policies`。
它只管推进逻辑。例如：
`连续两轮无关键信息增量时，必须 summarize_understanding 或 surface_conflict`
`核心问题 statement 未 grounded 前，不进入 requirements_draft_ready`
`同一轮最多 ask 两个问题`

第三组叫 `Evidence Policies`。
需求访谈里证据未必总来自工具，也可能来自用户原话、上传文档、历史需求条目。此类 policy 规定：
`用户原话可以 grounding fact slot`
`解释性总结只能写 proposed，直到用户确认或有文档支撑`
`外部文档引入的约束需要 evidence_ref`

第四组叫 `Schema Evolution Policies`。
这是这套设计里最特别的一层。它约束 controller 如何动态调整 InteractionIR：
`允许新增 assumption_slot`
`允许将新出现术语自动建 glossary artifact`
`允许将高频未决问题收敛成新 slot`
`禁止在同一轮同时新增超过 N 个 slot`

这些 policy 适合用 OPA/Rego 或等价规则系统实现，因为其输入天然是 JSON 状态，输出天然是 decision。OPA 官方文档明确支持把结构化数据作为 input，并对策略与执行做解耦。([Open Policy Agent][3])

然后是 LLM 驱动方式。为了让 controller 稳，我建议同一个 base LLM 分三个 prompt role 来跑。

`Interpreter Prompt`
职责是生成 patch proposal。它只看最新增量信息与相关局部状态，输出结构化 patch，不输出自然语言回复。

`Planner Prompt`
职责是在 allowed_acts 内选 act。它看局部状态摘要、策略输出、question budget，输出 chosen_act 与排序理由摘要。

`Realizer Prompt`
职责是把 chosen_act 渲染为实际访谈者说的话。它不允许私自新增 act，也不允许写入状态。

这样做的原因很现实。需求访谈里理解、计划、表达这三件事的失误模式不同。把它们拆开以后，错误更容易定位，运行日志也更清楚。文档里已经区分了 state update、policy check、action selection，这个三 prompt 设计正好把它具体化。

如果只针对需求访谈场景，我建议 controller 的 act selection 优先级采用一个显式打分函数。可以定义为：

`Score(act) = 信息增益 + 冲突消解收益 + checkpoint 推进收益 + 用户负担惩罚 + 风险惩罚`

其中：
信息增益高，优先
能解决冲突，优先
能推进 checkpoint，优先
一次问太多，扣分
会引入大量新 schema 扩展，扣分

这部分由 LLM 做语义估计，policy 只负责硬边界。这样能很好解释为什么 controller 有时先问目标用户，有时先澄清术语，有时先总结理解。它是在 allowed_acts 集内做上下文化排序。

这套 controller 的核心价值在于，它没有把需求访谈能力塞进 agent 的固定 prompt，也没有试图提前写死整棵对话树。它把需求访谈能力收敛成三类可复用机制：
`InteractionIR 的 profile`
`policy 的 decision logic`
`LLM 对局部状态的受约束读写`
这和文档里提出的“compile constraints, not dialogue”完全对齐。

最后给一个面向需求访谈的最小 controller 定义。

`Controller State Contract`
读取 `slot_state / issues / checkpoints / recent_acts`

`Controller Write Contract`
只能提交 `patch ops + chosen_act + checkpoint transitions`

`Controller Decision Contract`
只能输出 `decision / allowed_acts / required_evidence / tool_plan`

`Controller LLM Contract`
LLM 可以解释、排序、表述
LLM 不能绕过 policy 直接提交非法状态
LLM 不能生成 schema 外 act
LLM 可以在 extension policy 允许时提议新增 slot 或 issue

如果把它压成一句工程定义：

LLM-driven policy-guided interaction controller 就是一个“以 InteractionIR 为唯一状态源、以 policy 为唯一约束源、以 LLM 为语义解释与动作选择引擎”的回合式控制器。它对通用 agent 的增强点不在任务知识本身，增强点在于可检查的状态维护、可约束的动作选择、可追踪的交互演化。
