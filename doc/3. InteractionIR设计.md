目前仅关注InteractionIR本省这个交互状态中间件的设计，不上升到policy-guided interaction controller或skill等控制层。就其本身而言分析这个InteractionIR的结构（slots、policies、dialogue acts、tool recipes、checkpoints）。

把 InteractionIR 限定为“交互状态中间件”后，它的职责边界会清楚很多。它表达的是交互对象、交互状态、消息语义、能力引用、证据挂接、阶段快照这些“可持久化结构”；下一步该做什么、允许谁做什么、何时触发什么动作，这些属于控制层。这个切分与现有协议演化方向是对齐的。A2A 把任务、状态、消息、工件、能力声明分开建模；MCP 也把资源、工具、prompts、elicitation、lifecycle 作为协议层对象，而不是把控制器逻辑写进数据模型。([A2A Protocol][1])

就本体设计而言，把 InteractionIR 定义成一个“面向多方交互的有状态语义容器”，其顶层至少包含七个部分：interaction meta、parties、slots、acts、artifacts、policies、checkpoints。提到的 slots、policies、dialogue acts、tool recipes、checkpoints 都保留，但都降到“状态描述与约束描述”的位置，不承担控制器职责。A2A 里 message 与 artifact 分离、task 与 task state 分离，这个分离方式很值得吸收，因为它能避免把沟通内容、执行状态、结果载体混在一起。([A2A Protocol][1])

第一层是 interaction meta，也就是交互实例本身的身份与边界。这里至少要有 interaction_id、protocol_version、interaction_type、scope、created_at、updated_at、parent_interaction、related_task、retention_policy。原因很直接，现有 agent 协议都把版本、能力、会话边界、状态持久化当作基础设施要求。A2A 专门定义了协议版本、任务对象、任务状态与历史；MCP 明确要求 stateful connections、capability negotiation、lifecycle。InteractionIR 若想成为中间件，顶层元数据必须能支撑序列化、回放、迁移和跨系统交换。([A2A Protocol][1])

第二层是 parties。这个对象比 human/agent 标签更重要。每个 party 至少应有 party_id、party_type、role、capabilities_ref、authority_scope、visibility_scope、identity_ref、trust_level、preferred_modalities。W3C AI Agent Protocol 社区组把 inter agent communication、agent identity、standardized metadata formats、security and privacy mechanisms 视为同一组基础问题；A2A 也把 capability declaration 放进 AgentCard，用于后续交互协商。由此可见，参与方建模在交互层就是一等公民。没有这一层，后面的 acts、slots、tool references 都会失去语义锚点。([W3C][2])

第三层是 slots。slots 在 InteractionIR 里应当被定义为“交互过程中逐步填充的语义状态位”，而不是表单字段集合。一个 slot 至少要有 slot_id、semantic_type、cardinality、status、value、value_source、owner_party、audience、confidence、updated_at。这里的 status 建议单独标准化，例如 empty、proposed、grounded、contested、frozen、waived。A2A 的 INPUT_REQUIRED 这类状态说明，现实交互里“信息尚未补齐”本身就是协议层状态；MCP 的 elicitation 也说明“请求额外信息”应被显式建模。当前文档里的 provenance 思路应该直接下沉到 slot 级别，因为交互状态中最关键的不是值本身，是值当前处于什么确定性水平、来自谁、能否公开给谁。([A2A Protocol][1])

如果只看结构，slots 最好再细分成四类。context slots 表示任务背景与交互上下文。commitment slots 表示谁承诺了什么。evidence slots 表示外部来源绑定。preference slots 表示偏好、限制、风格之类的软状态。这样做的好处是，slot 系统能同时覆盖 human-AI 与 agent-agent。前者主要填 context 和 preference，后者会大量使用 commitment 和 evidence。这个分类来自文档里“信息要求、来源要求、修订状态、证据约束”已经混合出现的现实。

第四层是 dialogue acts。这里建议把它当成“消息语义类型系统”，不要把它设计成步骤或动作建议。A2A 已经表明消息会承担 task initiation、clarification、status update、ongoing interaction 等不同语义角色，而且输出结果由 artifact 承载。InteractionIR 可以借鉴这种分层，把 act 定义成消息的语义标签与结构契约。一个 act 至少包含 act_id、act_type、sender、receivers、about_slots、payload_schema、intent_ref、requires_response、creates_artifact、updates_slots、causal_parent、timestamp。这样，ASK_SLOT、PROPOSE_VALUE、REQUEST_EVIDENCE、SUBMIT_ARTIFACT、CHALLENGE_VALUE、ACK、DECLINE、CANCEL、HANDOFF 这些 act 就成为状态转换事件的“语义外壳”。([A2A Protocol][1])

这里有个关键点，dialogue acts 在 IR 层最好保持“可判定语义”，少放策略色彩。比如 ASK_SLOT 指明被询问的 slot，PROPOSE_VALUE 指明候选值与证据引用，CHALLENGE_VALUE 指明争议对象与理由，SUBMIT_ARTIFACT 指明附带结果对象。这样做有两个价值。第一，历史交互可以被压缩成 act log 后重建状态。第二，外部协议映射会简单很多，因为 A2A 的 message、artifact、task state 天然能对应到 act、artifact、checkpoint snapshot。([A2A Protocol][1])

第五层是 tool recipes。若只看中间件本体，我会把它定义成“能力引用模板”或“交互可调用外部接口的语义说明”，少强调 orchestration。MCP 区分了 resources、tools、prompts、elicitation，A2A 又强调交互双方无需暴露内部状态、记忆和工具，只需交换可协作所需信息。因此 InteractionIR 中的 tool recipe 更适合表示：此交互可引用哪些外部能力、能力输入槽位是什么、输出落到哪些 slot 或 artifact、能力结果的证据形式是什么。一个 recipe 至少包含 recipe_id、capability_ref、input_binding、output_binding、result_schema、evidence_schema、visibility、cost_class。([A2A Protocol][1])

这样定义后，tool recipe 在 IR 层承担的是“交互对能力的可声明接口”。它记录调用某类外部能力后，交互状态可能获得什么新信息、这些信息进入哪个 slot、返回物长什么样、可见范围多大。至于何时调用、调几次、失败后怎么补救，这些都留给控制层。这个切分和 MCP 的角色分离很一致，MCP 自身只定义工具与资源如何暴露和协商，不负责上层任务策略。([Model Context Protocol][3])

第六层是 policies。这次明确说不要上升到 controller，所以这里的 policies 最适合定义成“交互状态不变量、合法性条件、可见性规则、兼容性规则”，不写成 obligation 和 runtime decision rule。A2A 与 MCP 都把能力声明、状态约束、安全与授权当成协议层的一部分；W3C AI Agent Protocol 也把 identity、security、metadata 和 communication protocol 放在同一层讨论。由此看，InteractionIR 里的 policies 应该偏 schema like invariants。([W3C][2])

具体可分四类。第一类是 integrity policies，例如某些 slot 的值必须引用 artifact 或 evidence。第二类是 visibility policies，例如某个 slot 或 artifact 只对指定 parties 可见。第三类是 compatibility policies，例如某类 act 只能引用某些 payload schema。第四类是 lifecycle policies，例如某些 checkpoint 快照必须包含哪些 slot 状态。注意这里仍然只是“约束声明”。它表达什么状态算合法，什么绑定关系成立，什么视图可被暴露，不表达谁下一步必须做什么。这样，policies 与 controller 可以彻底解耦。

第七层是 checkpoints。只看 IR 本身时，checkpoint 最好理解成“状态快照的命名视图”，不把它定义成门控节点。A2A 的 task state 给了一个很强的提示，协议层确实需要表达当前交互处于 submitted、working、input_required、completed、failed 之类的阶段；MCP 也显式有 lifecycle。InteractionIR 可以借鉴这个思路，把 checkpoint 定义成 interaction state 的语义标签加校验面。一个 checkpoint 至少有 checkpoint_id、label、entry_condition_ref、required_slot_states、required_artifacts、frozen_fields、snapshot_hash、timestamp。([A2A Protocol][1])

这样设计后，checkpoint 的作用有三点。第一，它给交互过程提供语义化分段。第二，它允许跨系统交换时只传递某个阶段的压缩状态。第三，它为后续控制层提供干净输入，但自身保持中立。比如“requirements_collected”“evidence_grounded”“proposal_submitted”“handoff_ready”都可以是 checkpoint 名称。它们描述的是当前交互已经具备的状态形态。([A2A Protocol][1])

把这些拼起来，InteractionIR 的核心结构可以写成下面这种形式。

```json
{
  "interaction": {
    "meta": {},
    "parties": [],
    "slots": [],
    "acts": [],
    "artifacts": [],
    "tool_recipes": [],
    "policies": [],
    "checkpoints": [],
    "views": []
  }
}
```

其中 views 很有必要单列。因为同一个 interaction 可能要面向不同 party 暴露不同状态切片，尤其在 agent-agent 场景里，visibility_scope 与 authority_scope 很重要，AGNTCY 和 W3C 的相关工作都把 identity、messaging、observability、security 视为需要并列建模的基础设施。([W3C][2])

如果再压缩成一句定义，我会把每个元素的本体角色定成这样。slots 是状态位，acts 是语义事件，tool recipes 是外部能力引用模板，policies 是状态与关系约束，checkpoints 是命名快照。这个五元组已经足够构成交互状态中间件的骨架。再往上的 allowed acts、question budgets、require review、deny publish、next recommended queries 都属于另一个层次。

因此，InteractionIR 最合适的形态不是流程图，也不是消息协议的薄包装。它更像一个“可交换、可回放、可校验、可投影的交互状态模型”，中心是状态对象与语义事件，外围挂接能力引用、可见性规则和阶段快照。这样的结构既能容纳 human-AI interaction，也能无缝扩展到 agent-agent interaction。

这版模板里，五个核心对象的职责这样固定。

`slots`
承载交互状态位。重点字段是 `status`、`value`、`value_candidates`、`provenance`、`audience`。
建议统一状态枚举为 `empty / proposed / grounded / contested / frozen / waived`。这样一来，交互中的“不确定值”“有争议值”“已冻结值”都能显式落库。

`policies`
只保留状态约束声明，不放控制决策。
`integrity` 用于合法性，`visibility` 用于可见性，`compatibility` 用于 act 和 payload 的匹配关系，`lifecycle` 用于 checkpoint 快照一致性，`provenance` 用于证据绑定要求。

`dialogue acts`
作为语义事件日志。
一个 act 只说明“发生了什么语义事件”，例如 `ask_slot`、`propose_value`、`challenge_value`、`submit_artifact`。它通过 `status_patch` 改写状态，通过 `causal_parent_act_id` 串起因果链。

`tool_recipes`
作为外部能力引用模板。
它声明“给什么输入，产出什么 artifact，回填哪些 slot，生成哪些 evidence”。A2A 与 MCP 都把工具、资源、状态、工件分开表达，这正好支持把 recipe 压在 IR 层做静态声明。([A2A Protocol][2])

`checkpoints`
作为命名快照。
它记录某个阶段的状态切片，包含允许的 slot 状态、冻结字段、关联 act 与 artifact。A2A 的 task lifecycle 和 stateful task 对这种阶段化快照非常接近，MCP 也要求显式 lifecycle 与 related task 关联。([A2A Protocol][1])
