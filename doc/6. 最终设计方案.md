# InteractionIR 与 Controller 统一架构设计方案

## 1. 架构哲学与核心边界

本设计方案的核心目标是构建一个既支持**单 Agent + Human** 交互，又支持 **Multi-Agent (Party-Party)** 协作的统一控制器架构。

其核心哲学在于：**InteractionIR 并非一段写死的对话流程，而是一个自包含的、可演化的“状态中间件与领域契约”**。LLM 驱动的对话能力不直接内置于 Agent 的底层逻辑中，而是由 Controller 根据 InteractionIR 实时计算出**动态约束（Decision Packet）**与**动作目标（Chosen Act）**，再将其映射给外部 Agent 执行。

整个系统严格划分为三个层级：

1. **第一层：InteractionIR（中间件状态模型与领域契约）**
   * **角色**：系统的“数据库”与“宪法”。提供状态容器、动作类型系统、策略规则库，以及不可违背的状态完整性底线。
   * **特征**：它是 Controller 独占的内部状态机，外部实体不直接读写。
2. **第二层：Controller（动态约束生成器与状态维护者）**
   * **角色**：系统的“大脑”与“裁判”。
   * **特征**：读取 InteractionIR，推导本轮交互的边界（允许做什么、禁止做什么），生成对外部 Agent 生效的动态 Policy Packet 与 Chosen Act，并在校验合法后将外部输入 Patch 到 InteractionIR 中。
3. **第三层：External Agent（受限执行者 / 前台演员）**
   * **角色**：执行实际访谈与沟通的“嘴和手”。
   * **特征**：接收 Controller 输出的约束与动作目标，在有限的约束空间内发挥 LLM 的自然语言处理能力，生成具体话术，完成沟通。

## 2. InteractionIR：静态定义与状态模型

InteractionIR 的结构不再是单一的 JSON 模板，而是由 **Kernel Schema（稳定内核）**、**Domain Profile（场景配置）** 和 **Runtime State（运行时实例）** 三部分叠加而成。

### 2.1 Policy 与 Act 的静态形态 (Schema/Base)

在 InteractionIR 层面，Policy 和 Act 以**静态定义（Schema）**的形式存在，定义了领域内的规则空间：

* **Act Schema（动作类型系统）**：
  定义该场景下合法的语义动作集合。
  *例如：`ask_fact`, `clarify_term`, `summarize_understanding`, `surface_conflict`, `delegate_task`。*
* **Policy Schema / Base（策略规则库）**：
  定义约束的维度、业务准则和状态完整性底线。
  *例如：*
  * *状态完整性约束：`grounded` 状态的 Slot 必须携带 `source_refs`。*
  * *交互推进策略：连续两轮信息无增量必须触发 `summarize_understanding`。*
  * *预算控制：同轮提问 `ask_fact` 数量不超过 2。*

### 2.2 Runtime State 的核心对象

* **Parties（参与方）**：统一 Human、Agent、Tool Proxy，赋予角色（role）、权限（authority_scope）和能力引用。
* **Slots（状态位）**：交互中逐步形成的语义节点。
  * 关键字段：`status` (empty/proposed/grounded/contested/frozen)、`candidates` (记录协商冲突)、`provenance` (证据来源)、`extension_policy` (固定或允许 Controller 动态新增)。
* **Issues / Assumptions（动态问题追踪）**：记录访谈中发现的新问题、临时假设或范围冲突。
* **Artifacts vs Evidence（工件与证据）**：区分生成的外部化结果（Artifact）与支撑状态的依据（Evidence）。
* **Checkpoints（阶段快照）**：命名状态视图，如 `scope_baselined`，用于控制交互的阶段性推进。

## 3. Controller：组件流与动态约束生成

Controller 是一个以 InteractionIR 为唯一状态源、以 Policy 为唯一约束源、以 LLM 为语义解释引擎的**回合式控制流**。

### 3.1 Controller 的六大组件流

1. **State Interpreter（状态解释器，LLM驱动）**
   * **输入**：外部 Agent/Human 的输入、当前局部 State。
   * **输出**：`Patch Proposal`（提议更新 Slot 值、新增 Issue 等）。它只理解输入，不直接落库。
2. **Patch Validator（补丁校验器，Policy/Rule驱动）**
   * **职责**：依据 InteractionIR 的 **State Integrity Constraints** 拦截非法操作（例如试图将无来源的猜测直接写入 `grounded`）。
3. **Policy Evaluator（策略评估器，Policy/Rule驱动）**
   * **职责**：读取 Profile 中的 Policy Schema，结合当前状态，计算出对外部 Agent 生效的**动态约束（Decision Packet）**。
4. **Act Planner（动作规划器，LLM驱动）**
   * **职责**：在 Evaluator 给定的 `allowed_acts` 空间内，挑选出当前优先级最高的一项或多项动作，生成 **Chosen Act**。
5. **Response Realizer（响应渲染器，LLM驱动）**
   * **职责**：将枯燥的 Chosen Act 和 Decision Packet 翻译/组装成面向外部 Agent 的**强约束 Prompt**（或 API Payload）。
6. **State Committer（状态提交器）**
   * **职责**：将校验通过的 Patch、本轮生成的 Act 日志原子化地写入 InteractionIR。

### 3.2 Policy 与 Act 的动态形态 (Instance)

Controller 运行后，将静态的 Schema 实例化为**面向外部 Agent 的控制包**：

* **Decision Packet（动态策略实例）**：
  这层是外部 Agent 这一轮真正被约束的东西。
  ```json
  {
    "decision": "require_input",
    "allowed_acts": ["clarify_term", "ask_fact"],
    "blocked_acts": ["freeze_slot"],
    "focus_slots": ["slot:target_user"],
    "question_budget": 1,
    "notes_for_agent": ["优先消解术语歧义，避免引入新话题"]
  }
  ```
* **Chosen Act（动态动作实例）**：
  定义“这一轮具体要做什么”。
  ```json
  {
    "act_type": "clarify_term",
    "about_slots": ["slot:problem_statement"],
    "goal": "澄清“平台”指内部系统还是客户产品"
  }
  ```

## 4. 架构运行示例：从输入到外部动作的闭环

1. **外部输入**：Human 说：“系统主要是内部运营用，要接 ERP。”
2. **Interpret & Validate**：Controller 的 Interpreter 将其解析为 Patch（建议更新 `target_user`，新增 `ERP_integration` Issue）。Validator 检查合法后暂存。
3. **Evaluate & Plan**：Controller 发现有新的 Issue，触发 Policy 规则，Evaluator 限制下一步只能 `ask_fact`。Planner 决定动作目标为“澄清 ERP 接口细节”。
4. **Realize & Output**：Controller 将内部决定转译为对外部 Agent 的 Prompt：“你现在的任务是澄清 ERP 接口提供方，只问这一个问题，语气专业简短。”
5. **External Agent 执行**：外部 Agent 接收 Prompt，生成自然语言：“明白，请问目前 ERP 是否有现成的接口文档？”
6. **Commit**：Controller 将状态变更写入 InteractionIR。

---

## 5. 总结

本设计彻底解耦了**“状态、规则、控制逻辑与执行动作”**：

* **InteractionIR** 保障了数据的绝对合法性与通信契约的自描述性。
* **Controller** 作为内部的隐式脑，确保了对大模型“幻觉”和偏离轨道的拦截。
* **External Agent** 仅作为受限的前台演员，其行为完全受 Controller 下发的动态约束与动作目标驱动。

这套架构使得 InteractionIR 不仅能胜任复杂的单机需求访谈，更能平滑扩展为多智能体网络中的核心通信中间件标准。
