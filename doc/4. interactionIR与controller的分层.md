InteractionIR应该是一个交互状态中间件的设计，而policy-guided interaction controller是这个中间件的控制层。

在需求访谈场景，根据用户的输入，policy-guided interaction controller可以更新/维护InteractionIR，还可以决定下一次由LLM扮演的访谈者应该执行什么动作。
相当于给通用的agent套上一个policy-guided interaction controller之后，其就具有了需求访谈的能力。
值得注意的是，在这个场景里，InteractionIR是做对应的调整以适应需求访谈场景的，例如slot与动作等。
当然这个InteractionIR也不是一成不变的，是可以随着访谈进行，在规范指导下，由policy-guided interaction controller对齐进行动态调整/更新/维护的。

InteractionIR 负责“交互状态的表示、存储、演化和对外交换”。
policy-guided interaction controller 负责“读取当前状态、决定下一步动作、生成对状态的更新提案、提交更新”。
需求访谈能力来自 controller 对通用 agent 的约束与驱动；需求访谈场景适配则体现在 InteractionIR 的 profile 化，也就是 slot、act、artifact、checkpoint 这些对象按场景特化。这个划分与文档里“compile constraints, not dialogue”“stateful information model + obligations + tool policies + checkpoints”的主线是一致的。

按这个思路，InteractionIR 最合适的定义可以写成三层。

第一层是稳定内核。
这一层跨场景复用，长期稳定，主要包含：
`meta`
`parties`
`slots`
`acts`
`artifacts`
`evidence`
`checkpoints`
`views`
`history`

第二层是场景 profile。
这一层负责把同一个内核投到需求访谈场景，定义：
哪些 slot 类型存在
哪些 act 类型合法
哪些 artifact 重要
哪些 checkpoint 有意义
哪些字段允许动态扩展
哪些字段必须冻结

第三层是实例状态。
这一层是某一次具体访谈过程中的运行时状态，包括：
当前 slot 值
候选值与冲突值
已执行 acts
已收集 artifacts/evidence
当前 checkpoint 快照
状态变更历史

所以，InteractionIR 最自然的形态不是单一 JSON 模板，而是：

`Kernel Schema + Domain Profile + Runtime State`

这三个部分叠在一起，才构成真正的交互状态中间件。

如果把它进一步形式化，我建议 InteractionIR 本体只保留五类核心语义对象。

1. `Slot`
   表示“访谈中需要逐步形成的语义状态位”。

需求访谈里，slot 至少分四类：
`fact_slot`
项目背景、现状、约束、目标用户、预算、时间线这类事实位

`decision_slot`
优先级、取舍、范围边界、验收标准这类需要协商收敛的位

`assumption_slot`
暂定假设、待确认项、开放问题

`trace_slot`
来源、责任人、确认时间、证据链接

每个 slot 建议至少带这些字段：

```json id="w12bqs"
{
  "slot_id": "slot:target_user",
  "name": "target_user",
  "kind": "fact_slot",
  "semantic_type": "text | enum | set | number | date_range | uri | json",
  "cardinality": "one | optional | many",
  "status": "empty | proposed | grounded | contested | frozen | waived",
  "value": null,
  "candidates": [],
  "confidence": 0.0,
  "source_refs": [],
  "updated_by": "party:controller",
  "updated_at": "2026-04-20T12:00:00Z",
  "extension_policy": "fixed | extensible | controller_proposed",
  "notes": null
}
```

这里最关键的字段有三个。

`status`
它表达这个 slot 当前处在什么认知状态。需求访谈不是简单填表，很多信息会经历“提出、修正、争议、冻结”的过程。

`candidates`
需求访谈里大量信息一开始并不收敛，候选值必须保留，不能只存最终值。

`extension_policy`
这决定 controller 是否可以在访谈中增补新 slot。前面说 InteractionIR 可动态调整，这个能力就该落在这里。

2. `Act`
   表示“本轮交互发生了什么语义事件”。

在需求访谈场景，act 的集合应明显区别于 internal comms 之类场景。更合适的一组是：
`ask_fact`
`ask_constraint`
`ask_priority`
`clarify_term`
`propose_interpretation`
`summarize_understanding`
`surface_conflict`
`suggest_tradeoff`
`confirm_requirement`
`mark_unknown`
`add_assumption`
`revise_slot`
`freeze_slot`
`open_issue`
`close_issue`

每个 act 不只是消息文本，它还应携带对状态的影响：

```json id="86y0vk"
{
  "act_id": "act_01J...",
  "act_type": "propose_interpretation",
  "sender_party_id": "party:interviewer_agent",
  "recipient_party_ids": ["party:user"],
  "about_slots": ["slot:target_user", "slot:primary_goal"],
  "payload": {
    "text": "当前理解是内部运营同事为主，核心目标是减少手工汇总时间。"
  },
  "state_effect": {
    "slot_updates": [
      {
        "slot_id": "slot:target_user",
        "op": "propose_value",
        "value": "internal operations staff"
      }
    ],
    "issue_updates": []
  },
  "timestamp": "2026-04-20T12:03:00Z"
}
```

这样一来，InteractionIR 可以靠 act log 重建状态演化过程。文档里一直强调 interaction 要靠 stateful model 支撑、不能写成 rigid tree，这种 evented 设计正好匹配。

3. `Artifact / Evidence`
   表示“访谈过程中产生的外部化结果和支撑材料”。

需求访谈里常见 artifact 有：
访谈摘要
需求条目草案
术语表
问题清单
范围边界表
优先级矩阵
待验证假设列表

evidence 则包括：
用户原话摘录
上传文档片段
历史需求文档
业务规则说明
接口说明
外部链接

这两者要分开。artifact 是当前交互生成物，evidence 是状态依据。文档里已经把 provenance、tool result、links presence 看得很重，这个区分非常必要。

4. `Checkpoint`
   表示“当前交互进入了哪种命名状态快照”。

需求访谈里比起 publish，更自然的 checkpoint 是：
`context_seeded`
`core_problem_aligned`
`scope_baselined`
`constraints_grounded`
`open_issues_logged`
`requirements_draft_ready`
`requirements_review_ready`

checkpoint 在中间件层只存快照，不做决策。
它至少应记录：

```json id="f8p1xn"
{
  "checkpoint_id": "cp:scope_baselined",
  "label": "scope_baselined",
  "timestamp": "2026-04-20T12:20:00Z",
  "slot_snapshot": {
    "slot:problem_statement": "grounded",
    "slot:target_user": "grounded",
    "slot:non_goals": "proposed",
    "slot:timeline": "grounded"
  },
  "artifact_refs": ["artifact:req_outline_v1"],
  "issue_refs": ["issue:pricing_model_unknown"]
}
```

controller 之后可以基于这些 checkpoint 做 allow、block、ask、summarize 之类动作。checkpoint 自身只表达状态事实。

5. `Issue / Assumption`
   这是需求访谈场景里建议额外提升为一等对象的部分。

前面已经强调 InteractionIR 要能随着访谈动态调整。现实里动态变化大多通过三种东西发生：
发现新问题
提出临时假设
识别范围冲突

这些若只塞进 notes，会丢失结构。建议独立建模：

```json id="n1v8ql"
{
  "issue_id": "issue:integration_scope",
  "kind": "open_question | conflict | assumption | risk",
  "title": "是否需要纳入第三方系统集成",
  "status": "open | mitigated | resolved | waived",
  "related_slots": ["slot:scope", "slot:external_dependencies"],
  "source_refs": ["act_01J..."],
  "resolution": null
}
```

这会让 InteractionIR 更像真正的访谈状态中间件，因为需求访谈的核心从来都包含“问题管理”。仅有 slots 不够。

基于这些对象，我建议真实的 InteractionIR 顶层结构长成这样：

```json id="p7m4cx"
{
  "interaction": {
    "interaction_id": "int:req_interview:01J...",
    "schema_version": "interactionir/v0.2",
    "kernel_version": "core/v1",
    "profile_id": "requirements_interview/v1",
    "meta": {},
    "parties": [],
    "slot_schema": [],
    "slot_state": [],
    "act_schema": [],
    "acts": [],
    "artifacts": [],
    "evidence": [],
    "issues": [],
    "checkpoints": [],
    "views": [],
    "history": {
      "patch_log": [],
      "act_order": [],
      "checkpoint_order": []
    }
  }
}
```

这里特意把 `slot_schema` 和 `slot_state` 分开。
这是这个分层里最重要的一个实现点。

`slot_schema`
描述当前 interaction profile 下，系统承认哪些 slot、它们的类型与扩展策略

`slot_state`
描述本次访谈实例里，这些 slot 现在的值、状态、候选值、来源

因为前面已经明确说，需求访谈过程中 InteractionIR 会动态调整。那就意味着 schema 本身也会变，至少会发生：
新增 slot
拆分 slot
合并 slot
修改 slot 的 status
把某个 slot 从 optional 提升为 required like important
把某个 slot 冻结

如果 schema 和 state 不分开，动态维护会非常混乱。

所以我建议 controller 对 InteractionIR 的更新也不要直接“改对象”，最好统一成 patch 语义。
也就是 InteractionIR 自带一组允许的变更操作：

```json id="cf6g7e"
{
  "patch": {
    "patch_id": "patch_01J...",
    "proposed_by": "party:controller",
    "timestamp": "2026-04-20T12:10:00Z",
    "ops": [
      {
        "op": "add_slot",
        "path": "/slot_schema/-",
        "value": {
          "slot_id": "slot:success_metric",
          "kind": "decision_slot",
          "semantic_type": "text",
          "cardinality": "optional",
          "extension_policy": "fixed"
        }
      },
      {
        "op": "update_slot_status",
        "path": "/slot_state/slot:timeline/status",
        "value": "grounded"
      },
      {
        "op": "add_issue",
        "path": "/issues/-",
        "value": {
          "issue_id": "issue:data_source_unknown",
          "kind": "open_question",
          "status": "open"
        }
      }
    ],
    "justification": "用户新提到成功标准，但数据来源仍不明确"
  }
}
```

这样设计以后，那种“controller 在规范指导下动态调整/更新/维护 InteractionIR”就有了明确机制。
也就是说，InteractionIR 本身除了保存状态，还定义了“状态可如何被合法改写”。

再往前走一步，需求访谈 profile 最好再给出一个最小版本。可以像这样：

```json id="w4r2jm"
{
  "profile": {
    "profile_id": "requirements_interview/v1",
    "slot_schema": [
      {"slot_id": "slot:problem_statement", "kind": "fact_slot", "semantic_type": "text", "extension_policy": "fixed"},
      {"slot_id": "slot:target_user", "kind": "fact_slot", "semantic_type": "text", "extension_policy": "fixed"},
      {"slot_id": "slot:primary_goal", "kind": "decision_slot", "semantic_type": "text", "extension_policy": "fixed"},
      {"slot_id": "slot:constraints", "kind": "fact_slot", "semantic_type": "json", "extension_policy": "extensible"},
      {"slot_id": "slot:scope", "kind": "decision_slot", "semantic_type": "text", "extension_policy": "extensible"},
      {"slot_id": "slot:non_goals", "kind": "decision_slot", "semantic_type": "list_text", "extension_policy": "extensible"},
      {"slot_id": "slot:success_metrics", "kind": "decision_slot", "semantic_type": "list_text", "extension_policy": "controller_proposed"},
      {"slot_id": "slot:assumptions", "kind": "assumption_slot", "semantic_type": "list_text", "extension_policy": "controller_proposed"}
    ],
    "act_schema": [
      "ask_fact",
      "ask_constraint",
      "clarify_term",
      "propose_interpretation",
      "summarize_understanding",
      "surface_conflict",
      "suggest_tradeoff",
      "confirm_requirement",
      "mark_unknown",
      "add_assumption",
      "freeze_slot"
    ],
    "checkpoint_schema": [
      "context_seeded",
      "core_problem_aligned",
      "scope_baselined",
      "requirements_draft_ready"
    ]
  }
}
```

这一版已经能很好地体现核心观点：

InteractionIR 是通用中间件
需求访谈只是一个 profile
controller 读取 profile 和当前 state
controller 可以在 profile 允许的边界内动态改写 schema/state
LLM 驱动的访谈行为通过 controller 投射到通用 agent 上，形成“需求访谈者”能力

最后，把这些思想压成一句最准确的表述：

InteractionIR 应当被设计为“可 profile 化、可增量演化、可事件回放、可 patch 更新的交互状态中间件”；需求访谈能力不内置在 agent 本身，需求访谈能力来自 LLM-driven policy-guided interaction controller 对这个中间件的持续读写与动作选择。这个 controller 维护的是 state 与 schema 的对齐，驱动的是访谈轮次中的下一语义动作。文档里关于 stateful information model、dialogue acts、tool recipes、checkpoints、不要编译成 rigid dialogue tree 的判断，正好支撑这一分层。
